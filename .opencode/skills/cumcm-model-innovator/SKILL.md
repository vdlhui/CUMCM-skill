# P2：模型创新优化

## 角色分配
- 🟢 建模手主导：不足分析、创新模式匹配、公式链推导、可行性评估、反模式规避
- 🟡 编程手辅助：评估创新在代码层面的可实现性
- 🟡 论文手辅助：用叙事语言预写创新描述

## 输入
- P0 输出：problem_analysis.json（问题特征/数据条件/约束/隐含假设）
- P1 输出：model_route.json（选定模型链/三维评分/基线）
- 已注入知识库：references/model-innovation-patterns.md（22 种创新模式 + 决策树 + 5 种反模式）

## 执行步骤

### 第 1 步：标准模型不足分析

对 P1 选定的模型链中的每个环节，逐条分析不足：

```
1. 列出该模型在标准形式下依赖的假设
2. 对照 P0 的 data_plan 和 constraints，检查每条假设是否成立
3. 标记不成立的假设 → 这就是创新发力点

示例输出：
"标准 LSTM 假设所有时间步等权重，但 P0 数据画像显示：
 焊接温度在峰值时段(占数据 15%)的方差是平稳时段(占 85%)的 4.2 倍。
 等权处理会导致模型被平稳数据主导，忽略关键转折点。"
```

### 第 2 步：创新模式匹配

走 innovation-patterns 决策树：

```
题目数据量 > 5000 ?
  ├─ 是 → INN-01(特征工程)
  └─ 否 → 样本量 < 200 ? → INN-02(数据增强)

单一模型拟合优度预估 < 0.75 ?
  └─ 是 → INN-05(组合) + INN-06(注意力)

题目约束与实际有差距 ?
  └─ 是 → INN-11(现实约束) + INN-14(假设修正)

模型求解预估 > 30min ?
  └─ 是 → INN-15(启发式改进)

标准评价指标无法反映题目核心关切 ?
  └─ 是 → INN-19(新评价指标)
  └─ 否 → INN-20(交叉验证) + INN-22(置信度量化)
```

选择 1-2 个创新方向，不超过 3 个（叠Buff = 反模式 ANTI-03）。

### 第 3 步：完整公式链推导

对选定的每个创新方向：

```
1. 写出标准模型的公式（编号 + LaTeX）
2. 指出标准公式在本题场景下的不足
3. 写出创新后的公式（编号 + LaTeX）
4. 标注两式的关键差异
5. 给出"这意味着..."的自然语言解释
```

**示例**：

```
标准 LSTM 的隐藏状态序列输出为：
  H = [h₁, h₂, ..., h_T]                          (5)
其预测值通常取最后一帧或均值，所有时间步等权。

但本题温度序列中，峰值附近的温度变化率是平稳段的 4.2 倍（见 P0 数据画像）。
等权处理会使模型被平稳数据主导，忽略关键转折点。
为此引入时间注意力机制：

  α_t = (exp(e_t)) / (∑_{τ=1}^{T} exp(e_τ))       (6)
  e_t = v^T tanh(W_h h_t + W_s s_{t-1})           (7)
  c = ∑_{t=1}^{T} α_t h_t                          (8)

其中 α_t 是第 t 个时间步的注意力权重，由当前隐藏状态 h_t 和全局摘要 s 共同决定。
这意味着模型可以自动判别哪些时间步对最终预测更重要，
实验中将通过注意力权重可视化（P5a 高光图）直观展示模型学习到了哪些关键时间点。
```

### 第 4 步：四维可行性评估

对每个创新点打分（1-5）：

| 维度 | 评估问题 | 5 分标准 |
|------|----------|----------|
| **贴题性** | 创新是否直接解决了题目中的某个具体难点？ | 可引用题目原文佐证 |
| **可解释性** | 能否用"为什么有效"的直觉解释，而不是"因为实验效果好"？ | 有物理/统计/几何直觉支撑 |
| **数据支撑** | P0 中是否有足够数据支持创新的实现和验证？ | 数据量/维度/噪声水平满足要求 |
| **可复现性** | 创新是否依赖随机种子/特定硬件/封闭数据？ | 不依赖随机性，可在通用硬件复现 |

总分 ≥ 16 → 可行；总分 12-15 → 可行但需标注风险；总分 < 12 → 放弃此创新。

### 第 5 步：反模式扫描

逐一检查 5 种反模式：

- ANTI-01 炫技不贴题：创新是否匹配题目核心难点？
- ANTI-02 创新无验证：是否为创新设计了消融实验？
- ANTI-03 叠Buff：总创新数目是否 ≤ 2？
- ANTI-04 砍脚适履：创新方向是否与数据特征/问题结构自然匹配？
- ANTI-05 引入不可解：创新后模型复杂度是否在 P1 评估的可解范围内？

任何反模式命中 → 直接放弃该创新方向。

### 第 6 步：创新性自评估

```
创新级别：[增量改进 / 方法迁移 / 结构创新]
依赖假设：[列出创新成立的假设]
验证需求：[消融实验设计 / 对比实验设计 / 敏感性测试项]
预估独立贡献：[创新预计带来的指标提升幅度 + 理由]
```

### 第 7 步：生成结构化输出

汇总以上结果，生成 model-innovation.json：
- 创新条目列表（每个含：不足分析/创新模式/公式链/可行性评分/自评估）
- 标准模型vs创新后的公式对比表
- 反模式扫描结果（通过/命中及处理）

## 输出

| 文件 | 格式 | 内容 |
|------|------|------|
| `model-innovation.json` | JSON | 创新条目、不足分析、公式链(LaTeX)、四维可行性评分、自评估、反模式扫描结果 |

## 不能做的事
- 不要提出"无法用公式表达的创新"（如"更合理的参数选择"——那不是创新，那是调参，P4 的事）
- 不要提出没有消融实验设计的创新（ANTI-02）
- 不要同时提出 ≥ 3 个创新方向（ANTI-03）
- 不要无故绕开创新模式库——你走的每一个创新决策，都要能追溯到 decision tree 的某个节点
- 不要引入 P1 没有评估过复杂度的创新方案（否则会给 P3 埋红灯）
