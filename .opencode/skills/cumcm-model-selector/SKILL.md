# P1：模型方向选型

## 角色分配
- 🟢 建模手主导：模型链搜索、三维评分、路线决策
- 🟡 编程手辅助：评估每条链的实现成本（代码复杂度/时间/依赖）
- 🟡 论文手辅助：记录候选模型链名称和技术描述

## 输入
- P0 输出：problem_analysis.json / data_plan.json
- 已注入知识库：
  - references/common-model-catalog.md（80+ 模型目录 + 元模型选择表）
  - references/model-chain-patterns.md（45 条模型链 + 决策树 + 链质量检查）
  - references/model-success-rates.md（历年成功率统计 + 死亡组合表）

## 执行步骤

### 第 1 步：加载知识库
- 读取 model-catalog → 建立"任务类型→可用模型"映射
- 读取 model-chain-patterns → 建立"套路→候选模型链"映射
- 读取 model-success-rates → 获取当前套路的成功率排名

### 第 2 步：决策树匹配
基于 P0 的 pattern_match 和 task_type，走 model-chain-patterns 的决策树：

```
按 tasks_type 和 pattern_match.primary_pattern 定位模型链类别→
  从该类别的 45 条链中筛选 2-3 条候选 →
    匹配数据条件（样本量/特征数/序列长度）→
      排除不满足数据条件的链
```

### 第 3 步：候选链定稿
确定 2-3 条候选模型链，每条含：
- 链结构（各环节名称和连接关系）
- 适用性判断（与 P0 数据条件/约束/输出要求的匹配度）
- 引用的模型（从 model-catalog 中提取参数规模/适用场景/前提条件）

### 第 4 步：三维评分

对每条候选链，从三个维度打分（每维 0-100，含评分理由）：

| 维度 | 评分依据 | 高分条件 |
|------|----------|----------|
| **预期得分(Score)** | 该模型链在历年同类赛题中的获奖频率+当前题目匹配度 | 成功率统计"高"+"上升趋势"+与题目数据/约束高度匹配 |
| **实现成本(Cost)** | 代码复杂度×数据预处理成本×调参时间 | 评分越低越好（0=10分钟，100=12小时） |
| **风险(Risk)** | 数据不足风险+收敛失败风险+参数敏感风险 | 评分越低越好（0=零风险，100=高概率失败） |

**参考依据**：
- Score 参考 model-success-rates.md 中同套路模型链的获奖频率
- Cost 由编程手根据代码复杂度预估
- Risk 根据模型的前提条件是否满足、数据是否充分来判断

### 第 5 步：排除"死亡组合"
检查候选链是否命中 model-success-rates.md 中的"死亡组合"：
- 近 5 年 0 篇国一 → 直接排除
- 评委直接判低分 → 直接排除
- 替换为成功率统计中同类型的高成功率链

### 第 6 步：路线决策

按性价比优先原则推荐：

```
推荐最优链：Score 高 + Cost 中 + Risk 低
备选链(A)：Score 最高 + Cost 高 + Risk 中（需时间充裕）
备选链(B)：Score 中 + Cost 低 + Risk 低（保守方案）
```

决策理由必须包含：
- 为什么选这个而不是备选 A
- 在什么条件下应该切换到备选（剩余时间<Xh / 数据发现不满足前提）

### 第 7 步：生成结构化输出
调用 `scripts/build_model_route.py`：
- 将以上结果汇总为 model_route.json
- 包含：链结构、三维评分、基线模型、验证计划、备选方案

## 输出

| 文件 | 格式 | 内容 |
|------|------|------|
| `model_route.json` | JSON | 选定模型链/候选链(≥2)/三维评分(含理由)/基线模型/验证计划/备选方案/决策理由 |

## 不能做的事
- 不要跳过决策树直接凭直觉选模型
- 不要忽略"死亡组合"——即使看起来适合，成功率统计已证明不可用
- 不要只给评分不给理由——三维评分的每项必须有量化依据
- 不要选参数量级远超数据量的模型（如 200 样本选 Transformer → 红灯预埋）
