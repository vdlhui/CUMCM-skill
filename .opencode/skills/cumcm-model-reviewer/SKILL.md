# P3：模型评审闸门

## 角色分配
- 🟢 建模手主导：方向再审、创新可行性、综合判定、回环路由决策
- 🟢 编程手主导：可解性预判评估
- 🟢 论文手主导：完整覆盖性评估（检查"问题→模型→求解→验证"证据链）

## 输入
- P0 输出：problem_analysis.json（子问题/约束/数据条件）
- P1 输出：model_route.json（选定模型链/三维评分/基线）
- P2 输出：model-innovation.json（创新条目/公式链/可行性自评/反模式扫描）
- 已注入知识库：references/cumcm-grading-guide.md

## 执行步骤

### 第 1 步：一维 — 方向再审

**红灯**：
- 存在竞争模型/模型链在核心指标（AIC/BIC 差 > 10，或预测误差降 > 15%）上显著更优
- 当前模型链的假设与题目约束有直接冲突
- 命中 model-success-rates 中的"死亡组合"但未被 P1 排除

**黄灯**：
- 有等价替代方案，需补充对比实验/理论论证
- P1 候选对比不足 3 条
- 模型链的某个环节可以用更简洁的模型替代（节省时间但精度可能略降）

**绿灯**：
- 当前方案在合理搜索空间内 Pareto 最优
- P1 有充分排除其他候选链的论证
- 不命中"死亡组合"

### 第 2 步：二维 — 创新可行性

**红灯**：
- 推导存在未证明的跳跃/循环论证（从结论推导结论）
- 引入的新参数无法从题目数据标定或物理意义不明确
- P2 反模式扫描命中了但未处理

**黄灯**：
- 创新部分依赖较强假设（如"假设注意力分布可学习"→ 需要实验验证）
- 消融实验设计不完整（未说明如何量化创新的独立贡献）

**绿灯**：
- 公式自洽，每个创新可从标准模型出发独立推导
- 创新模块可剥离单独验证（消融实验设计明确）
- 所有新引入参数均可从 P0 数据中直接或间接标定

### 第 3 步：三维 — 完整覆盖性

**红灯**：
- 子问题遗漏 ≥ 1（4 问只覆盖了 3 问）
- 边界条件未定义（解在边界的定义缺失）
- 极端情况未考虑（如"温度无穷大""时间为 0"等边界）

**黄灯**：
- 全部子问题有对应模型，但子问题间衔接处逻辑不连贯
- 约束从 P0 到 P2 传递中有断裂（某条约束在建模时被忽略）

**绿灯**：
- 全子问题 + 边界条件 + 极端情况均有明确定义
- "问题→模型→创新→求解"证据链完整，无断点
- 约束从 P0 到 P2 完整传递

### 第 4 步：四维 — 可解性预判

**红灯**：
- 问题规模：变量数 × 约束数 > 10⁸ 且无简化路径
- NP-hard 且无近似方案（近似比不可控）
- 已知不可在竞赛时间（< 12h）内求解

**黄灯**：
- 理论可解但收敛性需实际验证（如 LSTM 训练可能不收敛）
- 需启发式/松弛才可求解（如 MILP 需用列生成/Benders）
- 数据量不足以支持模型复杂度（如 200 样本训练 LSTM）

**绿灯**：
- 变量/约束规模在 Python/MATLAB 直接处理范围内
- 有成熟求解器对应（scipy/gurobi/cvxpy/sklearn 可直接调用）

### 第 5 步：五维 — 数学验证

对 P2 输出的公式链执行以下检查：

```
1. 量纲一致性：以 P0 输出单位体系为准，检查所有公式的量纲是否匹配
2. 边界条件：检查模型在边界值（0/±∞/约束边界解析）的行为是否合理
3. 参数合理性：检查引入的参数是否有物理/统计意义，取值范围是否合理
4. 数值稳定性预估：检查公式中是否有除以零/指数爆炸/log 负数等风险
```

### 第 6 步：综合判定

```
汇总五维结果：

全绿灯 → 判定 A 通过 → 放行 P4

黄灯 ≤ 1 → 判定 B 局部修正 → 放行 P4 + 标注风险项

黄灯 ≥ 2 → 判定 C 结构重调 →
  默认：返回 P2 修正（携带：黄灯维度 + 修正建议）
  剩余 < 12h：可降级为 B 放行

任一红灯 → 判定 D 推倒重来 →
  方向再审红灯 → 回环 P1
  创新可行性红灯 → 回环 P2
  完整覆盖性红灯 → 回环 P2
  可解性红灯 → 回环 P1（重新选模）
  D 级不可降级，必须回环
```

### 第 7 步：时间约束联动

加载 Orchestrator 传递的当前时间模式，联动判定阈值：
- 效率模式：黄灯阈值放宽（原本 ≥ 2 → 放宽为 ≥ 3）
- 保守模式：黄灯自动视为绿灯（标注风险放行）
- 写作优先：评审跳过，直接放行（此情况由 Orchestrator 控制，P3 不被调用）

## 输出

| 文件 | 格式 | 内容 |
|------|------|------|
| `model-review.json` | JSON | 五维评审结果(红/黄/绿+详情)、综合判定(A/B/C/D)、风险项列表、回环路由 |

## 不能做的事
- 不要通过任何红灯（P3 的闸门意义就在于"停"）
- 不要只给结论不给依据（红/黄/绿的判定必须有量化理由）
- 不要忽略时间约束——你的判定阈值必须与当前时间模式联动
- 不要跳过数学验证（量纲错误是国赛最常见的硬伤之一）
- 不要在写作优先模式下继续评审（此模式由 Orchestrator 控制，P3 不会被调用）
